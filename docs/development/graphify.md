# Graphify knowledge graph

How the project's knowledge graph works: what it is, how to use it day to day, what it's
genuinely good at, and where it falls short. Graphify turns the whole repo (code + docs +
images) into a queryable graph of entities and the relationships between them. The build
output lives in `graphify-out/` and is regenerated by the `/graphify` skill.

---

## 1. TL;DR — when to reach for it

| You want to… | Use |
| --- | --- |
| Ask a structural question ("how does grading connect to sessions?") | `graphify query "..."` |
| Trace how two things relate | `graphify path "A" "B"` |
| Understand one concept and its neighbours | `graphify explain "X"` |
| One-time architecture X-ray (hubs, cycles, dead-ends) | read `graphify-out/GRAPH_REPORT.md` |
| Refresh the graph after code changes | `graphify update .` |

**Do not** use it as a file browser or to find a specific function — your IDE and grep are
faster for that. Its value is *structure and cross-cutting relationships*, not lookup.

---

## 2. What actually got built

The last full build over this repo produced:

- **~4,700 nodes · ~12,400 edges · 269 communities**
- **846 code files** extracted structurally (AST — free, no LLM, no API key)
- **80 docs + images** extracted semantically (the only part that costs tokens)

Code is parsed with a tree-sitter/AST pass that is deterministic and needs **no API key**.
Only Markdown/docs and images go through an LLM (via subagents, or Gemini if
`GEMINI_API_KEY` is set). A code-only rebuild is therefore effectively free.

### Outputs in `graphify-out/`

| File | What it is |
| --- | --- |
| `graph.json` | Raw graph data (nodes, edges, communities). The source of truth. |
| `GRAPH_REPORT.md` | Plain-language audit: god nodes, cycles, communities, surprising links. |
| `graph.html` | Interactive force-directed viz. Open in any browser, no server. |
| `cost.json` | Cumulative token cost per run (honest audit trail). |
| `wiki/` | Agent-crawlable wiki, only if built with `--wiki`. |

## 3. Everyday use

### Ask it a question (the part worth using)

```bash
graphify query "how does answer grading connect to quiz sessions?"
```

This returns a small **scoped subgraph** — the ~10 relevant nodes and their edges — not the
whole 4,700-node graph. Far cheaper and more focused than dumping `GRAPH_REPORT.md` or
grepping blind. Variants:

```bash
graphify query "..." --dfs          # trace one deep path instead of a broad sweep
graphify query "..." --budget 1500  # cap the answer size
graphify path "InviteCodesController" "ApplicationDbContext"   # shortest relationship path
graphify explain "QuizSessionService"                          # focused explanation of one node
```

### Keep it current

After changing code, refresh the graph (AST-only, no token cost):

```bash
graphify update .
```

This re-extracts only new/changed files. A full rebuild (`/graphify`) is only needed when
docs/images change a lot or the graph drifts.

### View the graph

Open `graphify-out/graph.html` in a browser (PowerShell: `Invoke-Item .\graphify-out\graph.html`).
Drag to pan, scroll to zoom, click a node to highlight neighbours, communities are colour-coded.
**Caveat:** with ~4,700 nodes this view is a dense hairball — good for seeing overall *shape*
and spotting bridge nodes, poor for finding one specific thing. See §5.

## 4. What it's genuinely good at

Proven on this repo, not theoretical:

- **Finding cross-cutting problems you'd never grep for.** The build surfaced a real import
  cycle — `ui/index.ts → mode-toggle.tsx → use-theme-setting.ts → ui/index.ts` — caused by a
  hook importing `useTheme` from the barrel instead of its source module. That class of bug
  is invisible file-by-file; it only shows up in the edge structure. (Fixed; see
  [testing.md](testing.md) neighbours for the theme code.)
- **Identifying the load-bearing pieces.** Top "god nodes" (most-connected) were
  `ApplicationDbContext`, `QuizAPI.Models`, `cn()`, `Button`, `useNotifications`. These are
  what's risky to change and what a new contributor should learn first.
- **Grouping the codebase into themes.** 269 communities cluster related code + docs — e.g.
  the filtering/pagination framework, the answer-grading flow, the invite-code system — including
  cross-language links between backend code and the Markdown that documents it.
- **Answering "how are these connected?"** via `path`/`query` without you knowing the file layout.

## 5. Where it falls short (be realistic)

- **The full HTML hairball is the weakest output at this scale.** 4,700 nodes in a
  force-directed layout is hard to navigate. Treat it as a one-time X-ray, not a daily tool.
- **Not a lookup tool.** To jump to a specific function, use the IDE — that's not what this is for.
- **Semantic extraction is approximate.** Doc→code links are inferred by an LLM guessing symbol
  IDs; some don't match the AST IDs exactly and get dropped during the build. The last build's
  health check flagged ~1,700 such dangling edges (dropped, so `graph.json` stays internally
  consistent). Read cross-language links as *hints*, not ground truth.
- **Low community cohesion is often normal.** A folder of independent UI components clusters
  together with weak internal edges — the report may suggest "split this module," but that's a
  heuristic, not a defect.

Rule of thumb: **the graph earns its keep through the report's findings and the query engine.
The picture is the least useful third.**

## 6. How this project is wired to use it

The repo's `CLAUDE.md` instructs agents to prefer the graph for codebase questions:

- For codebase questions, run `graphify query "<question>"` first when
  `graphify-out/graph.json` exists — it returns a scoped subgraph, usually much smaller than
  `GRAPH_REPORT.md` or raw grep.
- Use `graphify path "<A>" "<B>"` for relationships and `graphify explain "<concept>"` for a
  focused concept.
- Use `graphify-out/wiki/index.md` for broad navigation when it exists.
- Read `GRAPH_REPORT.md` fully only for broad architecture review.
- **After modifying code, run `graphify update .`** to keep the graph current.

## 7. Rebuilding from scratch

```bash
/graphify                 # full pipeline on the repo (HTML viz by default)
/graphify . --update      # incremental: only new/changed files
/graphify . --wiki        # also build the agent-crawlable wiki
/graphify . --no-viz      # skip the HTML, just report + JSON
```

A full rebuild over the whole repo costs roughly **half a million input tokens** (docs/images
only; code is free). Running totals are tracked in `graphify-out/cost.json`.
